# Week 3 — Quiz

Ten multiple-choice questions. Take it with your lecture notes closed. Aim for 9/10 before moving to Week 4. Answer key at the bottom — do not peek.

---

**Q1.** The six-hour solo build is best described as:

- A) A small product launch, intended to get users.
- B) A rehearsal — the rhythm of "demo or die" practiced alone, with the retro as the real artifact.
- C) A portfolio piece for job applications.
- D) A coding speed-test against other learners.

---

**Q2.** Your one-sentence prompt is, "A social platform for sharing pet photos." According to Lecture 1 §2.1, this prompt is:

- A) Fine; it names a topic.
- B) Over scope — it has no single user, no single verb, and no nameable pain.
- C) Too narrow; it should mention monetization.
- D) Acceptable if you skip auth.

---

**Q3.** On a six-hour solo build's MoSCoW chart, MUST should contain:

- A) Five features so you have options.
- B) Three features (one frontend, one backend, one infra).
- C) Exactly one demo-able feature, plus the "live deploy URL" line.
- D) Whatever your partner suggests.

---

**Q4.** The hourly cut reflex (Lecture 1 §5) is performed:

- A) Once at the start of the build, to plan the day.
- B) Every 60 minutes, on the hour, out loud, with four written one-line answers.
- C) Whenever you feel stuck.
- D) Only if you fall behind on MUST.

---

**Q5.** At hour 5:55, you have a broken deploy and a half-finished change. The correct action is:

- A) Push the half-finished change; the judge will understand.
- B) Roll back to the last green commit, ship that, name the change in the retro.
- C) Skip the recording — the demo will not help anyway.
- D) Extend the clock by 30 minutes.

---

**Q6.** Which of the following is **not** a behavior, by the Lecture 2 §4.3 test?

- A) "At hour 0:00, set a kitchen-timer with a loud ringtone."
- B) "Be more disciplined about deploys."
- C) "At every hourly stand-up, curl the deploy URL and write the HTTP status in the scratch file."
- D) "Write the demo script in the README by Wednesday."

---

**Q7.** The "ask" line of a Week 3 solo demo recording (Lecture 1 §7.3) is best aimed at:

- A) Investors interested in seed funding.
- B) Beta users who want to try the tool.
- C) The retro reader — "tell me one cut I should have made earlier."
- D) Your professor, for extra credit.

---

**Q8.** A retro bullet that reads "I am bad at scoping" is:

- A) Blameless and acceptable.
- B) Self-blaming and forbidden; translate to a system-level observation about the MoSCoW chart or the hourly check.
- C) The only honest way to write a retro.
- D) A goal, which is fine in a retro.

---

**Q9.** The hour-five freeze (Lecture 1 §6) means:

- A) You stop work entirely and rest until hour six.
- B) No new code paths, components, or routes; the last hour is for the demo path, README polish, recording, and the retro draft.
- C) Only frontend code is frozen; backend changes are fine.
- D) Only optional; the hour-five freeze is a suggestion.

---

**Q10.** Which is the **most accurate** statement about the three-behaviors-to-change list (Lecture 2 §4)?

- A) It is optional; some learners skip it.
- B) Three is too many — name one strong behavior and move on.
- C) Each behavior has a *when* (the time anchor) and a *what* (the specific action); the list is the retro's payload and grades 15% of the final capstone.
- D) Five behaviors is better than three; more is more.

## Answer key

<details>
<summary>Click to reveal answers</summary>

1. **B** — A rehearsal. The retro is where the learning lives, not the build. (Lecture 1 §1)
2. **B** — No user, no verb, no pain. Re-read the one-sentence prompt test. (Lecture 1 §2.1)
3. **C** — Exactly one MUST feature plus the "live deploy URL" line. Two MUSTs eat hour 5 every time. (Lecture 1 §3.1)
4. **B** — Every hour, on the hour, out loud, four questions, written one-line answers. The fourth question is the muscle. (Lecture 1 §5)
5. **B** — Roll back. The half-finished change does not exist for demo purposes; name it in the retro. (Lecture 1 §6.2)
6. **B** — "Be more disciplined" is a goal, not a behavior. A behavior has a *when* and a *what* and you could film yourself doing it. (Lecture 2 §4.3)
7. **C** — The retro reader. Solo demos point the "ask" at the retro, which is the artifact you actually want feedback on. (Lecture 1 §7.3)
8. **B** — Self-blame terminates inquiry. Translate to a system observation: the MoSCoW chart, the hourly check, the timer. (Lecture 2 §2.2)
9. **B** — No new code paths; the last hour is for demo path, README, recording, retro draft. (Lecture 1 §6)
10. **C** — Each behavior has a *when* and a *what*; the list is the retro's payload; the SYLLABUS capstone rubric grades it. (Lecture 2 §4 and SYLLABUS §"Capstone rubric")

</details>

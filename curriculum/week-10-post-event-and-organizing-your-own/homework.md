# Week 10 — Homework

Six practice problems for the week. Two on the *carry-out* (the attendee side), four on the *organising* (the org side). Each homework problem produces a small artifact — a one-paragraph write-up, a checklist, or a one-page draft. Homework is *applied work that lives outside the exercise file structure* — usually saved in a personal-notes file or a `homework/` folder in the team repo.

## Rubric

| Score | Criterion |
|-------|-----------|
| 5     | All six problems attempted; each produces a named artifact; each artifact is between the stated length floor and ceiling; each is committed or shareable |
| 4     | Five of six problems complete; remaining one started but partial |
| 3     | Four of six problems complete; some artifacts under-developed |
| 2     | Three of six problems complete; artifacts thin |
| 1     | Fewer than three problems attempted |
| 0     | Not submitted |

A score of 4+ is the expected target. 5 is achievable in the time budgeted (2 hours total).

---

## Problem 1 — The 24-hour-after thank-you message

Draft the *one-paragraph group thank-you message* the team's carry-out lead sends to teammates 24 hours after the closing ceremony. The message is part of Day 1 of the carry-out window (Lecture 1 §1).

**Constraints.** 60–120 words. Names every teammate. Names the next-step decision (who is owning the recruiter-grade README, who is uploading the phone-demo video, who is drafting the blog). Closes with a specific deadline (e.g., "the README pass is happening by Wednesday").

**Artifact.** Save as `homework/01-thank-you-message.md`. Commit to the team repo or save in a personal notes folder.

---

## Problem 2 — Re-write your existing project README opener

Take the *existing* README of any past project of yours (any project — does not have to be a hackathon). Re-write *only the first paragraph* using the C4 *problem-statement-first* convention from Lecture 1 §2.

**Constraints.** 3–5 sentences in the new opener. Names the user. Names the moment. Does not open with the tech stack. Saves both the old opener and the new opener side-by-side in the artifact file.

**Artifact.** Save as `homework/02-readme-rewrite.md`. Include both versions plus a 2-sentence reflection on what changed.

---

## Problem 3 — Walk the four-question gate for your own campus

For your own campus (the one where the capstone hackathon would be held), answer each of the four gating questions from Lecture 2 §1 *honestly* in writing.

**Constraints.** One paragraph per question. State *what you actually have right now*, not what you hope to have. The 4-question answers feed directly into the mini-project's *opening assessment* section.

**Artifact.** Save as `homework/03-four-gate.md`. The honest version of this document is the strongest possible input to the capstone.

---

## Problem 4 — Sketch a venue map

Sketch (paper-and-photo, or simple Figma, or just a labeled bulleted list) the *room you would use* for your capstone hackathon. Identify the named placements from Lecture 3 §5: hacking floor, opening-ceremony seating, food serving, restrooms, organisers' room, quiet zone, sponsor table area, judging table area.

**Constraints.** Each named placement *labelled*. Include the named *power outlet audit* count if you can do it (walk the room and count). Include the named *AV check* note (where the projector and screen sit).

**Artifact.** Save as `homework/04-venue-map.md` (with embedded photo or sketch) or as `homework/04-venue-map.png`. The artifact lives in the mini-project as well.

---

## Problem 5 — Pick your MLH tier and write the rationale

Decide which MLH membership tier (Local / Member / Premier / none) you would apply for for the capstone. Write the rationale.

**Constraints.** 200–400 words. Names the named tier benefits from Lecture 2 §3. Names the named cost (verify on `organize.mlh.io/`). Names the named *break-even attendee count* (Lecture 2 §3 cites ~80 attendees as the Local-to-Member break-even). States the named cost-benefit verdict for *your* capstone specifically.

**Artifact.** Save as `homework/05-mlh-rationale.md`. The decision feeds the mini-project's *MLH track* section.

---

## Problem 6 — Draft the opening-ceremony script

Draft the *opening-ceremony script* for the capstone hackathon. 45 minutes total (Lecture 3 §5). Five named blocks: welcome, code-of-conduct recital, theme + judging walk-through, sponsor intros, logistics walk-through, Q&A.

**Constraints.** Approximately one paragraph per block. The *full script* — what the lead organiser says aloud. Includes the named *code of conduct recital* (60–90 seconds, paragraph form). Includes the named *report-channel* mention by name. Total length: 600–900 words.

**Artifact.** Save as `homework/06-opening-script.md`. The script is rehearsable — the lead organiser reads it aloud once with a timer to confirm the 45-minute fit.

---

## Submission

Commit all six artifacts under `homework/` in the team repo (or in a personal notes folder if not yet a team repo). The mini-project (capstone) re-uses Problems 3, 4, 5, and 6 directly; the homework is *not throwaway*, it is *capstone groundwork*.

A note on time: 2 hours is the named budget. If you find yourself at 4 hours, you are over-engineering. The artifacts are *first drafts*; the mini-project is where the polish happens.

## Pairing with the capstone

The homework deliberately under-counts the capstone work. Reuse map:

- **Problem 1** → does *not* re-appear in the capstone; the thank-you message is a one-shot artifact for the carry-out week (per-team, not per-event).
- **Problem 2** → does *not* re-appear in the capstone; it sharpens the README sense that powers Exercise 1.
- **Problem 3** → re-appears verbatim as *Section 3* of the capstone's `PLAN.md`.
- **Problem 4** → re-appears verbatim as `mini-project/venue/venue-map.md`.
- **Problem 5** → re-appears verbatim as *Section 4* of the capstone's `PLAN.md`.
- **Problem 6** → re-appears verbatim as `mini-project/day-of/opening-ceremony-script.md`.

If you scope the homework to the capstone's reuse pattern, you avoid the named *double-effort failure mode* — writing the venue map twice, once for homework and once for the mini-project. Write it once; reference it from both places.

## A note on honesty

Problems 3, 4, and 5 are the *honesty* problems. The temptation in writing a hackathon plan is to write the version you wish were true: the venue we hope to get, the co-organisers we hope to recruit, the MLH tier we hope to afford. The C4 voice is firm on this: *write the actual answer, in writing, today*. The aspirational version of the plan is the named *self-deception failure mode*; it produces a document that looks good on paper and fails in week 5 of the runway. The honest version is the document that *gets to event day*.

If your honest answers to Problem 3 fail two or more of the four gates, your honest answer to the capstone is *not yet*. The C4 voice does not penalise *not yet*. The C4 voice penalises *pretending it is yet when it is not*.
